## 1. ОБЩАЯ АРХИТЕКТУРА ПРОЕКТА

Проект решает сложную задачу: как написать код вида `var z = x + y` и заставить классическую Машину Тьюринга (МТ) его вычислить? 
Для этого система разделена на следующие этапы:
1. **Лексический и синтаксический анализ:** Текст разбивается на токены, строится дерево абстрактного синтаксиса (AST).
2. **Распределение памяти:** Каждой переменной выделяется строгий участок на ленте 0.
3. **Генерация правил (Компиляция):** Проход по AST генерирует тысячи элементарных правил МТ (сдвинуть головку, прочитать символ, записать). 
4. **Выполнение (Симуляция):** Движок МТ принимает сгенерированные правила и пошагово меняет состояния лент.

Машина использует **3 ленты**:
- **Лента 0:** "Оперативная память". Хранит все переменные подряд.
- **Лента 1:** "Регистр A". Сюда копируется первый операнд для вычислений. Сюда же пишется результат.
- **Лента 2:** "Регистр B". Сюда копируется второй операнд.

## 2. РАЗБОР ИСХОДНОГО КОДА ПО ФАЙЛАМ

### `TuringMachine.h` / `.cpp`
**Суть:** Это "железо". Класс, который тупо и слепо исполняет правила.

**Ключевые структуры данных:**
```cpp
// Ленты представлены как std::map<int, char>. 
// Почему map? Потому что лента бесконечна в обе стороны (индексы могут быть отрицательными), 
// и map позволяет выделять память только под те ячейки, куда мы реально что-то записали.
std::vector<std::map<int, char>> tapes; 
std::vector<int> heads; // Позиции головок для 3 лент
```

**Правила (Rules):**
Правило определяет переход МТ: `(Текущее состояние, Символы_на_лентах) -> (Новое состояние, Что_записать, Куда_сдвинуть_головки)`.
```cpp
// Хранение правил. Ключ - имя состояния (например "q1"). 
// Значение - вектор структур Rule, так как из одного состояния может быть много переходов 
// в зависимости от того, что мы прочитали на лентах.
std::map<std::string, std::vector<Rule>> rules; 
```

**Метод `Step()` (Самый важный метод ядра):**
```cpp
bool TuringMachine::Step() {
    if (currentState == "halt") return false; // Остановка

    // 1. Читаем символы под головками всех трех лент
    char c1 = ReadTape(0); char c2 = ReadTape(1); char c3 = ReadTape(2);

    auto it = rules.find(currentState);
    if (it == rules.end()) return false;

    // 2. Ищем подходящее правило для текущих символов
    for (const auto& rule : it->second) {
        bool match = true;
        // '?' - это Wildcard (джокер). Он означает "мне неважно, какой тут символ, правило подходит".
        if (rule.readSymbols[0] != '?' && rule.readSymbols[0] != c1) match = false;
        if (rule.readSymbols[1] != '?' && rule.readSymbols[1] != c2) match = false;
        // ...
        
        if (match) {
            // 3. ПРИМЕНЯЕМ ПРАВИЛО
            for (int i = 0; i < 3; ++i) {
                // Записываем новый символ (если не '?')
                if (rule.writeSymbols[i] != '?') tapes[i][heads[i]] = rule.writeSymbols[i];
                // Сдвигаем головку (-1, 0, или 1)
                heads[i] += static_cast<int>(rule.moveDirections[i]);
            }
            currentState = rule.nextState; // Меняем состояние автомата
            return true;
        }
    }
    return false; // Нет правила - аварийная остановка (ошибка)
}
```
*Если преподаватель спросит:* "Зачем нужен символ `?` в `readSymbols` и `writeSymbols`?"
*Ответ:* Без джокера нам пришлось бы писать правила для каждой возможной комбинации символов на трех лентах. Джокер `?` при чтении означает "игнорировать, что на ленте", а при записи — "оставить символ на ленте без изменений".

### `MemoryManager.h` / `.cpp`
**Суть:** Организация переменных на Ленте 0.

**Архитектура памяти (VAR_SIZE = 34):**
Каждая переменная занимает строго 34 ячейки:
- **`[0]`** - Символ `#`. Служит разделителем. По нему головка понимает, где начинается новая переменная при поиске.
- **`[1]`** - Знак. `+` или `-`.
- **`[2]...[33]`** - 32 бита данных. Хранятся от старшего бита (MSB) к младшему (LSB).

**Метод `Deploy()`:**
Этот метод вызывается перед запуском МТ. Он берет начальные значения переменных из C++ и физически прописывает их на Ленту 0 в бинарном виде.
```cpp
void MemoryManager::Deploy(TuringMachine& tm) const {
    tm.SetTapeContent(0, -1, '!'); // Граница слева. Защита от ухода в минус бесконечность
    // ...
    for (const auto& var : variables) {
        int basePos = var.varIndex * VAR_SIZE;
        tm.SetTapeContent(0, basePos, '#'); // Ставим маркер начала переменной
        
        int val = var.initialValue;
        if (val < 0) {
            tm.SetTapeContent(0, basePos + 1, '-');
            val = -val;
        } else {
            tm.SetTapeContent(0, basePos + 1, '+');
        }

        // Преобразование числа в 32 бита
        for (int i = 0; i < 32; ++i) {
            int bit = (val >> (31 - i)) & 1; // Достаем i-тый бит числа
            tm.SetTapeContent(0, basePos + 2 + i, bit ? '1' : '0');
        }
    }
}
```

### `MacrosForTuring.h` / `.cpp`
**Суть:** В МТ нет функций и циклов `for`. Чтобы сделать цикл "иди вправо, пока не встретишь #", нужно сгенерировать несколько состояний. Здесь лежат "макросы" — функции C++, которые генерируют кучу правил для МТ, формируя сложные алгоритмы.

**Разбор `GenerateAlgorithmicSeek` (Как МТ находит переменную "y"?):**
Чтобы найти переменную под номером `varIndex`, МТ должна пройти вправо и отсчитать нужное количество символов `#`.
```cpp
void GenerateAlgorithmicSeek(TuringMachine& tm, const std::string& startState, int varIndex, const std::string& endState) {
    std::string curr = startState;
    for (int i = 0; i < varIndex; ++i) {
        std::string next = startState + "_seek_" + std::to_string(i);
        std::string skipCurrent = next + "_skip";
        
        // 1. Шаг с текущего '#' вправо, чтобы не топтаться на месте
        tm.AddRule(curr, '#', '?', '?', skipCurrent, '?', '?', '?', Direction::Right, Direction::Stay, Direction::Stay);
        
        // 2. ПРОПУСК: Идем вправо, пока видим 0, 1, + или - (игнорируем содержимое переменной)
        tm.AddRule(skipCurrent, '0', '?', '?', skipCurrent, '?', '?', '?', Direction::Right, Direction::Stay, Direction::Stay);
        // ... аналогично для '1', '+', '-'
        
        // 3. Если наткнулись на следующий '#', значит пропустили одну переменную
        if (i == varIndex - 1) {
            // Если это нужная переменная по счету - останавливаемся (переходим в endState)
            tm.AddRule(skipCurrent, '#', '?', '?', endState, '?', '?', '?', Direction::Stay, Direction::Stay, Direction::Stay);
        } else {
            // Иначе продолжаем поиск со следующего состояния
            tm.AddRule(skipCurrent, '#', '?', '?', next, '?', '?', '?', Direction::Stay, Direction::Stay, Direction::Stay);
            curr = next;
        }
    }
}
```

**Разбор сложения `GenerateBinaryAdd`:**
Перед сложением ленты 1 и 2 уже стоят на позиции 34 (на младших битах). Сложение идет справа налево (от LSB к MSB) с учетом переноса (`carry`).
```cpp
// Создаем два состояния: 
// add0 - мы сейчас складываем биты, и переноса в уме нет (carry = 0)
// add1 - мы складываем биты, и у нас есть 1 в уме (carry = 1)
auto addRules = [&](const std::string& state, int carry, const std::string& next0, const std::string& next1) {
    for (char b1 : {'0', '1'}) {
        for (char b2 : {'0', '1'}) {
            int val1 = b1 - '0'; // Чар в число (0 или 1)
            int val2 = b2 - '0';
            int sum = val1 + val2 + carry; // Складываем биты с лент и перенос из ума
            
            char res = (sum % 2) + '0'; // Что пишем на Ленту 1
            std::string next = (sum >= 2) ? next1 : next0; // В какое состояние переходим
            
            // Записываем res на Ленту 1, сдвигаем обе головки влево (к старшим битам)
            tm.AddRule(state, '?', b1, b2, next, '?', res, b2, Direction::Stay, Direction::Left, Direction::Left);
        }
    }
};
addRules(add0, 0, add0, add1); // Генерируем правила для carry=0
addRules(add1, 1, add0, add1); // Генерируем правила для carry=1
```
*Ключевой фикс:* Для сложения критически важно, чтобы головки 1 и 2 стояли на последнем бите числа. Для этого используется специальная функция `GenerateRewindTape0`, которая перематывает в начало только ленту 0 (поиск результата), не сбивая позиции головок на лентах 1 и 2 после их загрузки.

### `Compiler.h` / `.cpp`
**Суть:** Превращает текст в вызовы макросов.

**Как компилируется `var z = x + y` (метод `CompileAST`):**
1. Строится AST дерево: Корень `+`, левый ребенок `x`, правый `y`.
2. Рекурсивный вызов спускается к листьям (`x` и `y`).
3. Генерируется уникальное имя временной переменной для результата сложения: `_t_1`.
4. Далее компилятор генерирует цепочку состояний МТ, сшивая макросы:
```cpp
// 1. Поиск x на Ленте 0
std::string s2 = NextState();
GenerateAlgorithmicSeek(tm, s1, mem.GetVarIndex("x"), s2);

// 2. Копирование x с Ленты 0 на Ленту 1 (операнд A)
std::string s3 = NextState();
GenerateLoadTape0ToTape1(tm, s2, s3);

// 3. Перемотка ТОЛЬКО Ленты 0 в начало, чтобы искать y
std::string s4 = NextState();
GenerateRewindTape0(tm, s3, s4); 

// 4. Поиск y и загрузка на Ленту 2 (операнд B)
std::string s5 = NextState();
GenerateAlgorithmicSeek(tm, s4, mem.GetVarIndex("y"), s5);
std::string s6 = NextState();
GenerateLoadTape0ToTape2(tm, s5, s6);

// 5. Вызов бинарного сложения (Лента 1 + Лента 2 -> результат пишется на Ленту 1)
std::string s7 = NextState();
GenerateBinaryAdd(tm, s6, s7);

// 6. Сохранение результата: ищем _t_1 (или z) на ленте 0 и копируем в него данные с ленты 1
std::string s10 = NextState();
GenerateStoreTapeToTape0(tm, s9, 1, s10);
```

## 3. ЧАСТЫЕ ВОПРОСЫ НА ЗАЩИТЕ (Q&A)

**В: Почему переменная занимает ровно 34 ячейки, а не 33 или 35?**
О: 1 ячейка под маркер-разделитель `#`, 1 ячейка под знак `+` или `-`, и ровно 32 ячейки под 32-битное представление числа (int32). Итого: 1 + 1 + 32 = 34.

**В: У вас бесконечная лента. Как вы избегаете ошибки переполнения памяти в C++?**
О: Лента реализована не как массив `char[9999]`, а как хеш-таблица (дерево) `std::map<int, char>`. Память в ОЗУ компьютера выделяется только под те ячейки ленты МТ, в которые машина Тьюринга реально что-то записала. Пустые ячейки (возвращающие `_`) не существуют в памяти.

**В: Зачем нужны временные переменные (например, `_t_1`)?**
О: При сложном выражении, например `z = x + y + 5`, машина сначала должна вычислить `x + y`. Результат этого промежуточного вычисления нужно где-то сохранить на ленте 0 перед тем, как прибавить к нему `5`. Компилятор сам выделяет адрес под `_t_1` и использует его.

**В: Если я напишу `var x = 10`, где происходит перевод числа 10 в двоичную систему? Это делает Машина Тьюринга?**
О: Нет, это делает компилятор (программа на C++) перед запуском МТ в методе `MemoryManager::Deploy`. МТ оперирует уже готовыми битами на ленте. МТ — это вычислитель (АЛУ), а перевод из десятичной в двоичную — это задача среды выполнения перед стартом.

**В: Почему была проблема, что переменная `z` оставалась нулем после вычисления суммы, и как это было исправлено?**
О: Проблема была в функции `GenerateRewind`. После загрузки `x` на ленту 1 и `y` на ленту 2, головки этих лент стояли на 34-й позиции (на младших битах). Но перед загрузкой `y` вызывался полный Rewind, который возвращал головку 1-й ленты обратно на ноль. Алгоритм сложения `GenerateBinaryAdd` начинал идти влево от нуля, упирался в ограничитель ленты и завершал работу сбоем до того, как результат записывался в `z`.
Решение: я написал `GenerateRewindTape0`, которая перематывает только нулевую ленту, чтобы найти `y`, не трогая головку на первой ленте, где уже готов к вычислению операнд `x`.

**В: Можно ли расширить язык до поддержки `while` или `if`?**
О: Архитектурно — да. В `Compiler` нужно будет добавить узлы AST для ветвления. А в генерации правил добавить генерацию уникальных состояний-меток (по аналогии с `goto`). Компилятор будет генерировать правило: "Если на ленте 1 результат сравнения (0 или 1) равен 1, перейти в состояние метки тела цикла, иначе — в состояние конца цикла".
